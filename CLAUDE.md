# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

DisplayPad Configurator — a **client-only PWA** (SvelteKit + Svelte 5 + Tailwind 4) for connecting to and configuring a **Mountain DisplayPad** straight from the browser over **WebHID**. The DisplayPad is a 6×2 macro pad with a 102×102-pixel display under each of its 12 keys. There is no backend and no OS-level agent: the app talks to the hardware directly, paints the keys, and reacts to key presses in-page.

## Architecture

Four layers, low-level → UI. The key rule: the **protocol layer is pure and Node-testable**; only `device.ts`/`raster.ts`/`liveface.ts`/`sandbox.ts`/`template.ts`/`favicon.ts` may touch browser/hardware APIs (`template.ts`'s two string-transform helpers, `wrapHtmlInSvg`/`renderMustache`, are the exception — pure and unit-tested — the DOM only enters via its `rasterizeHtml`/`fetchTemplateFace` orchestration; likewise `favicon.ts`'s `hostnameFrom`/`faviconServiceUrl` are pure and unit-tested, browser APIs enter only via its `fetchFaviconDataUrl`).

1. **Protocol (pure)** — `src/lib/displaypad/protocol.ts`, `image.ts`
   The USB HID wire format, isolated so it can be unit-tested without a browser: `VENDOR_ID 0x3282` / `PRODUCT_IDS [0x0009]`, `NUM_KEYS` 12 in a 6×2 grid, `ICON_SIZE` 102 px square, images in **BGR** order (`encodeImage` flips RGBA→BGR), `PACKET_SIZE`-byte pixel payloads, the `INIT_MESSAGE`/`IMAGE_MESSAGE` control frames, and key-press decoding (`decodeKeyStates` reads bitmasks at report bytes 42 & 47).

2. **Device transport (browser-only)** — `src/lib/displaypad/device.ts`, `raster.ts`, `liveface.ts`, `sandbox.ts`, `template.ts`, `favicon.ts`
   `DisplayPad extends EventTarget`: `request()` / `fromGranted()` → `open()`; emits `keydown` / `keyup` / `close`. It drives the reverse-engineered transfer state machine (INIT → announce a key → pad echoes → stream pixel chunks) from inbound `inputreport` events. The pad is a **composite HID device with three interfaces** — see "Hardware caveats" below; `assignRoles()` splits the handles by report descriptor size, not enumeration order. WebHID delivers a report as `reportId` + `data`; `buildReport()` rejoins them so the protocol byte offsets line up (a no-op for the pad's unnumbered reports, where `reportId` is always 0 and `data` already starts at the real type byte). `raster.ts` turns an image URL (data URL, blob URL, or same-origin/CORS-permissive remote URL) into the 102×102 RGBA buffer the protocol expects. `liveface.ts`'s `fetchRemoteFace()` backs the `remote` `KeyFace`: it fetches a URL over GET, sniffs `content-type`, injects a `width`/`height` on SVG responses that lack one (canvas rasterisation is otherwise unpredictable), and hands the result to `rasterize()` as a blob URL — the endpoint must send CORS headers permissive to this origin or the `fetch` itself fails. `sandbox.ts` runs a `template` face's optional `transform` in an opaque-origin `sandbox="allow-scripts"` iframe (`runTransform`), talking `postMessage` with a hard timeout and structured-clone validation; `runOverChannel` is the pure message-contract half, tested against a mocked channel, while `createIframeChannel` is the real (untested) DOM plumbing. `template.ts`'s `fetchTemplateFace()` runs that transform (handing it `ctx = { now, secrets }`, the secrets passed in by the caller — see the `secrets` store below), feeds the result through Mustache (`renderMustache`, dynamic-imported), and rasterises the HTML via `rasterizeHtml()` — SVG `<foreignObject>` today, feature-detecting the not-yet-shipped native `drawElementImage` canvas API first. `favicon.ts` backs the "auto icon from an Open URL action" convenience (KeyInspector): its pure `hostnameFrom()`/`faviconServiceUrl()` build a [unavatar.io](https://unavatar.io) aggregator URL (favicon / app icon / social image, `fallback=false` so a miss is a clean 404, and CORS-permissive so the browser can actually read the bytes), and `fetchFaviconDataUrl()` fetches it into an `image`-face data URL — best-effort, returning `null` (never throwing) on a bad/absent URL, a 404, or a CORS/network failure. Auto-population only overwrites a never-touched default face; the explicit "Use site icon" button previews the fetched icon and asks before replacing any non-blank face. unavatar.io is credited in the page footer. Verbose `[displaypad]` wire tracing is gated behind the `debug` singleton (below) and a checkbox in the UI.

3. **State (Svelte 5 runes singletons)** — `src/lib/state/*.svelte.ts`
   - `keymap` — `pages: KeyConfig[][]` (each page 12 `KeyConfig`s: label / face / action) plus an `activePage` rune, persisted to `localStorage`. The source of truth for what each key looks like and does; hardware-agnostic. `keys` reads/writes `pages[activePage]`, so most of the app still treats it as a flat 12-key grid; `importAll(keys)` replaces the keymap with a single page, `importPages(pages)` with several. A `navigate` action (`{ type: 'navigate'; target: number | 'back' | 'next' }`) either jumps `activePage` to another page (`target` a page index — mirroring Base Camp's "Create Folder"), with `target: 'back'`, pops `pageHistory` to return (Base Camp's "Back"), or with `target: 'next'`, advances to `(activePage + 1) % pageCount` (wrapping around) via `openPage()` — a browser-only convenience with no Base Camp equivalent, so it round-trips through Base Camp export as unassigned with a warning, like `webhook`; `openPage()`/`back()`/`next()`/`switchPage()` manage that navigation and history. Legacy `open-folder`/`back` actions (and the removed `copy-text` action) are folded into the current union by `migrateAction`/`migratePages` (`$lib/types.ts`) on every keymap/template load and import. A `remote` face (`{ type: 'remote'; url; refreshMinutes?; refreshOnPress? }`) only persists the URL + policy — fetched pixels are transient and re-fetched each cycle. A `template` face (`{ type: 'template'; template; transform?; refreshMinutes?; refreshOnPress? }`) likewise only persists the HTML/Mustache template and optional sandboxed JS transform, never rendered pixels; `isLiveFace()` (`$lib/types.ts`) is true for both `remote` and `template`. `importPages()`/`importAll()` set `scriptsApproved = false` whenever the incoming pages carry a `template` face with a (non-empty) `transform` — a transform is executable JS, so an imported one must not run silently; `approveScripts()` is the explicit opt-in, surfaced as a banner in `ProfileTools.svelte`. Locally-authored edits (KeyInspector) never touch this flag.
   - `connection` — owns the single `DisplayPad`, mirrors pressed state into runes, pushes keymap faces to the hardware on explicit apply only (`applyKey` / `applyAll` — neither runs automatically on connect), and runs a key's configured action when it is pressed — including `navigate`, which calls `keymap`'s page navigation and then repaints via `applyAll()` (`goToPage()` / `goBack()` / `goNext()` / `jumpToPage()`, the last used by the UI's breadcrumb, not just physical presses). For a `remote` or `template` face, `applyKey()` calls `fetchRemoteFace()`/`fetchTemplateFace()` and records failures per-key in `liveFaceErrors` instead of throwing (CORS/network/transform failures on a live face must not break the rest of the pad) — a `template` face whose `transform` isn't yet approved (see `keymap.scriptsApproved` above) is refused the same way, without ever invoking the sandbox. `syncLiveTimer(index)` (re)derives that key's `setInterval` from the currently-shown face (a toggle key's `secondFace` while flipped, otherwise `face`) via `isLiveFace()` — call it after editing a key's face; the `keydown` handler also re-syncs it after a toggle flip, so a live second face refreshes on its own policy; it's a no-op while disconnected. Timers refetch+apply on a floor of 1 minute (with a little jitter) and are all cleared in `teardown()`, so a live face does no background traffic while disconnected. `refreshOnPress` is handled inline in the `keydown` listener. On construction it silently tries `DisplayPad.fromGranted()` to reopen a previously-permitted pad without prompting; `connect()` (the `ConnectButton` click handler) is the only path that calls `DisplayPad.request()` and shows the picker.
   - `debug` — one `enabled` boolean, toggled by a checkbox in `+page.svelte`, read by `device.ts`'s `debugLog()` to gate console tracing of the wire protocol.
   - `templatePreview` — renders a `template` face's current output for the in-browser "virtual keypad" tile (`PadKey`), independent of hardware connection. `scheduleRender(index, face, scriptsApproved, secretValues)` debounces (300ms) then calls `fetchTemplateFace()` (passing `secretValues` through to the transform's `ctx.secrets`) + `pixelsToDataUrl()`, storing the result in `images[index]`/`errors[index]`; a `template` face with an unapproved `transform` is refused the same way `connection.applyKey()` refuses it, without invoking the sandbox. On success it (re)schedules its own `refreshMinutes` timer (same 1-minute floor/jitter as `connection`'s), so a live preview (e.g. a clock/weather template) keeps refreshing on-screen even while disconnected. Takes `face`/`scriptsApproved`/`secretValues` as arguments rather than reading `keymap`/`secrets` itself — `PadKey`'s `$effect` reads them, which both supplies the current values and (as a side effect of Svelte's dependency tracking) is what triggers a re-render whenever any of them changes.
   - `secrets` — a persisted list of `{ key, value }` credentials (its own `displaypad.secrets.v1`, kept out of the keymap store) so they never ride along in a Base Camp export or a saved stash template — only the _reference_ travels. Referenced two ways: `{{secret.KEY}}` placeholders in a `webhook` action's header values and JSON body (resolved by `connection.fireWebhook()` via `secrets.apply()`), and a `ctx.secrets` object handed to a `template` transform (`applyKey`/`templatePreview`/`templates` pass `secrets.values` into `fetchTemplateFace()`), so a credential can authenticate a `fetch` without living in the stored/exported config. `substituteSecrets`/`isValidSecretKey` are pure and Node-tested; keys are identifiers so both forms stay unambiguous. Edited via `SecretsDialog.svelte`, opened from the overflow menu (under "Apply all on (re)connect") and from a "Secrets" button above each compatible field. Storage is plain-text and device-local — scoping, not encryption.

4. **UI** — `src/routes/+page.svelte` + `src/lib/components/*.svelte` (`ConnectButton`, `PadGrid`, `PadKey`, `KeyInspector`, `FaceEditor`, `CodeEditor`). Presentational only; they read and mutate the stores. `PadKey` is the one exception with any logic of its own: an `$effect` calls `templatePreview.scheduleRender()` for its **active** face (primary, or the `secondFace` while a toggle key is flipped) whenever that face, `keymap.scriptsApproved`, or `secrets.values` changes, keeping the tile's rendered preview current — every other mutation (edits, drag/drop swaps, stash applies, page navigation, Base Camp import) reaches it for free since it all flows through reassigning `keymap.keys[index]`, which is the prop `PadKey` already re-renders from. `FaceEditor.svelte` is the single editor for one face — the kind picker (color/image/remote/live) plus every type's inputs, the burned-on text label, and the live-refresh policy — used by `KeyInspector` for **both** the primary face and the toggle key's second face, so the two are configurable identically. Switching the kind picker is **non-destructive**: it only reveals that kind's editor (a `faceKindOverride` that diverges from `face.type` until a real edit lands), so flipping to "Color"/"Live" and back never wipes an existing image/remote/template config; the "Live" starter HTML shows only as the editor's _placeholder_ and isn't committed until typed. Every edit is committed through an `onchange` callback; `KeyInspector` wires that to `keymap.update({ face })` / `keymap.update({ secondFace })` + `connection.syncLiveTimer()`, wraps each `FaceEditor` in `{#key index}` so per-key local state resets on selection, and bumps a `resetNonce` on "Reset" to clear a pending reveal on the same key. `CodeEditor.svelte` wraps CodeJar + Prism.js (dynamic-imported, bundled — no CDN) for the template face's HTML/Mustache and transform JS fields, with an optional greyed-out `placeholder` overlay shown while empty.

### PWA / rendering

- **Client-only.** `src/routes/+layout.ts` sets `ssr = false` + `prerender = true`. adapter-static emits a prerendered shell that hydrates and then runs entirely in the browser (WebHID has no server equivalent).
- **Offline.** `src/service-worker.ts` (SvelteKit auto-registers it in production builds) cache-first precaches the app shell and static assets. The manifest link and `theme-color` are in `src/app.html`; assets are `static/manifest.webmanifest` and `static/pwa-icon.svg`.

## Conventions & gotchas

- **There is no `svelte.config.js`.** SvelteKit config (the static adapter and the forced-runes `compilerOptions`) lives inside the `sveltekit()` plugin call in `vite.config.ts`. Change SvelteKit options there.
- **Runes are forced on** for all app code (everything outside `node_modules`). Use `$state` / `$derived` / `$props`; the stores are plain classes in `.svelte.ts` files exported as singletons.
- **Relative imports use explicit `.js` extensions** (tsconfig `rewriteRelativeImportExtensions`). Use the `$lib` alias for `src/lib`.
- **Nothing touching `navigator.hid` or the DOM may run at module top-level or during SSR.** Device access is guarded via `DisplayPad.isSupported()` and `browser` from `$app/environment`.
- **WebHID typings** come from `@types/w3c-web-hid`, referenced from `src/app.d.ts`.

## Hardware caveats (validated 2026-07-09 against a physical pad)

The protocol is reverse-engineered; the pure layer is unit-tested and both the image-write path and key-press decoding (`keydown` firing a key's `open-url` action) have been confirmed end-to-end on real hardware.

- **Three HID interfaces, not two.** `navigator.hid.requestDevice()` grants: a generic consumer-control/keyboard interface (unused — 0-byte output reports), a **display** interface (single unnumbered output report, 1024 bytes — matches `CHUNK_SIZE`, receives pixel chunks only, no input reports at all), and a **control** interface (single unnumbered input+output report, 64 bytes — receives `INIT_MESSAGE`/`IMAGE_MESSAGE`, emits key-state/ack reports). `assignRoles()` picks display/control by these report-descriptor sizes; don't go back to enumeration-order guessing.
- **Unnumbered reports need `buildReport()` to pass through unchanged.** WebHID reports `reportId: 0` for the control interface's single implicit report, and `event.data` already starts with the real type byte (0x01/0x11/0x21) — nothing was stripped, so `buildReport()` must not prepend a synthetic 0 byte for `reportId === 0` (it only reconstructs for a genuinely nonzero, numbered `reportId`). Getting this wrong silently shifts every downstream offset — including the `KEY_BITS` byte-42/47 lookups — by one.
- **`INIT_MESSAGE`/`IMAGE_MESSAGE` are 65 bytes** (a leading pseudo report-id byte + a 64-byte HID report), matching the interrupt-endpoint size used by the reference drivers — not the doubled 112-byte constants an earlier version of this file had (silent no-op writes, no visible error).
